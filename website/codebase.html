Runtime Flow

The CLI and config create a run context, then the executor dispatches stage implementations through the 23-stage state machine.

01

CLI and Config

researchclaw/cli.py and config.py resolve projects, LLMs, runtime flags, HITL settings, servers, MCP, export, memory, and skills.

→
02

Run Context

pipeline/runner.py manages artifact directories, run IDs, checkpoints, heartbeats, resume behavior, and deliverable packaging.

→
03

Stage Contracts

pipeline/stages.py and contracts.py define phases, rollback rules, gates, required artifacts, outputs, retry budgets, and done criteria.

→
04

Executor

pipeline/executor.py validates inputs and outputs, applies HITL hooks, calls LLM providers, scores stages, and dispatches implementation modules.

Why It Matters

Downstream stages find inputs by contract names or by scanning stage directories. That keeps resume and repair flexible, but it also means artifact naming, best-result promotion, and package assembly need to remain stable.

Architecture Notes

Repo memory also tracks mismatches and edge cases that future changes should keep visible.

Version and packaging drift

The README describes newer v0.4.0-era features while pyproject.toml declares version 0.3.1. The wheel package list includes sibyl and arc, but matching source directories were not present during the scan.

Guarded optional modules

runner.py references optional modules such as researchclaw.pipeline.event_log and researchclaw.cost_tracker through guarded imports. Missing optional files degrade silently.

Gate semantics

QUALITY_GATE is both a gate stage and listed as noncritical, so blocking behavior depends on runner and executor flags.

Citation ordering

CITATION_VERIFY runs after EXPORT_PUBLISH, even though comments indicate citation checks are intended to block export of hallucinated citations.
